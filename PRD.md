# BOSS Auto Greeter - 产品需求文档 (PRD)

## 1. 背景

求职者在 BOSS 直聘平台上需要手动浏览大量岗位，逐个发送打招呼消息。这个过程耗时、重复且效率低下。特别是在求职高峰期，求职者每天需要花费数小时在这类机械性操作上。

同时，AI 大模型技术（如 SiliconFlow 提供的 API）已经能够生成高质量的个性化打招呼内容，但将这些能力与 BOSS 直聘平台结合需要一定的技术门槛。

## 2. 目标用户

### 主要用户画像
- **求职者**：正在寻找工作的技术人员、产品经理等白领人群
- **特点**：
  - 熟悉基本电脑操作
  - 有 BOSS 直聘账号
  - 愿意使用 AI 工具提升求职效率
  - 对数据隐私有一定要求（倾向于本地存储）

### 次要用户
- **招聘顾问**：需要在 BOSS 平台上主动触达候选人的 HR
- **职业转型者**：需要大规模投递简历的转行者

## 3. 产品价值

### 对用户的价值
- ⏱️ **节省时间**：自动化重复性操作，每天可节省 2-3 小时
- 🎯 **提升效率**：批量处理，单次可处理数十个岗位
- 💬 **提高质量**：AI 生成个性化打招呼内容，提升回复率
- 🔒 **保护隐私**：本地运行，数据不上传第三方服务器

### 对市场的价值
- 填补"BOSS 直聘自动化工具"的市场空白
- 降低 AI 技术使用门槛（用户只需提供 API Key）

## 4. 用户流程

### 首次使用流程
1. 下载并安装 Windows 桌面应用
2. 输入 SiliconFlow API Key（从 SiliconFlow 官网获取）
3. 扫码登录 BOSS 直聘账号
4. 配置求职偏好（岗位关键词、城市、薪资范围等）
5. 配置打招呼模板（可选，系统提供默认模板）

### 日常使用流程
1. 启动应用（自动加载上次的配置）
2. 输入搜索关键词和筛选条件
3. 点击"开始搜索"
4. 系统展示搜索结果（岗位列表）
5. 用户勾选目标岗位
6. 点击"批量打招呼"
7. 系统为每个岗位生成个性化打招呼内容
8. 预览并确认发送
9. 查看发送结果（成功/失败统计）

## 5. 功能需求

### 5.1 核心功能

#### F1 - BOSS 账号管理
- **F1.1 扫码登录**：通过 BOSS 直聘二维码扫码登录
- **F1.2 登录状态管理**：记住登录状态，避免重复扫码
- **F1.3 多账号支持**（V2）：支持切换多个 BOSS 账号

#### F2 - 岗位搜索
- **F2.1 关键词搜索**：输入岗位名称、公司名等关键词
- **F2.2 筛选条件**：
  - 城市（支持多选）
  - 薪资范围
  - 工作年限
  - 学历要求
  - 公司规模
- **F2.3 搜索结果展示**：列表形式展示，包含关键信息
- **F2.4 收藏功能**：标记感兴趣的岗位

#### F3 - AI 打招呼
- **F3.1 模板管理**：提供默认模板，支持自定义
- **F3.2 AI 生成**：调用 SiliconFlow API 生成个性化内容
- **F3.3 内容预览**：发送前预览生成的打招呼内容
- **F3.4 手动编辑**：允许用户修改 AI 生成的内容

#### F4 - 批量操作
- **F4.1 批量选择**：勾选多个岗位
- **F4.2 批量打招呼**：一键发送多个打招呼消息
- **F4.3 进度显示**：实时显示发送进度
- **F4.4 结果统计**：成功/失败数量及原因

#### F5 - API Key 管理
- **F5.1 Key 输入**：用户输入自己的 SiliconFlow API Key
- **F5.2 本地加密存储**：Key 存储在本地，加密保护
- **F5.3 余额查询**：显示 API 余额（可选）
- **F5.4 使用统计**：显示 API 调用次数和费用

### 5.2 辅助功能

#### F6 - 配置管理
- **F6.1 求职偏好配置**：保存默认搜索条件
- **F6.2 导入导出**：配置的导入导出（JSON 格式）

#### F7 - 历史记录
- **F7.1 打招呼历史**：查看已发送的打招呼记录
- **F7.2 回复状态**：跟踪 BOSS 是否回复（V2）

#### F8 - 系统设置
- **F8.1 主题切换**：浅色/深色模式
- **F8.2 语言切换**：中文/英文
- **F8.3 自动更新**：检测并提示新版本

## 6. 非功能需求

### 6.1 性能要求
- 应用启动时间 < 3 秒
- 单次搜索响应时间 < 5 秒
- 批量打招呼速度：10 条/分钟（避免触发反爬）
- AI 生成单条打招呼时间 < 3 秒

### 6.2 安全要求
- API Key 本地加密存储（AES-256）
- BOSS 登录凭证本地存储，不上传服务器
- 代码开源，接受社区审计
- 敏感操作需用户确认

### 6.3 可用性要求
- Windows 10/11 兼容
- 无需管理员权限安装
- 单文件可执行程序（可选）
- 友好的错误提示

### 6.4 稳定性要求
- 异常情况下不丢失用户数据
- 网络断开时提供清晰提示
- API 调用失败时自动重试（最多 3 次）

## 7. 风险与应对

### 7.1 技术风险
| 风险 | 影响 | 应对措施 |
|------|------|---------|
| BOSS 平台反爬机制 | 账号被封禁 | 1. 限制频率 2. 模拟人类行为 3. 明确提示风险 |
| BOSS 接口变更 | 功能失效 | 1. 监控接口变化 2. 快速迭代更新 |
| SiliconFlow API 限制 | 生成失败 | 1. 重试机制 2. 降级方案（使用模板） |

### 7.2 法律风险
| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 违反 BOSS 平台规则 | 账号封禁 | 1. 严格限制频率 2. 用户协议明确风险 |
| 数据隐私问题 | 法律纠纷 | 1. 所有数据本地存储 2. 不收集用户信息 |

### 7.3 用户风险
| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 用户过度使用 | 账号被封 | 1. 每日上限提醒 2. 智能限流 |
| API Key 泄露 | 费用损失 | 1. 本地加密存储 2. 定期提醒更换 |

❌ 历史记录查看

❌ 自动更新

### V1 目标

- 用户体验流畅

## 10. 成功指标

- 日活用户数（DAU）
- 平均每日打招呼数量

- 应用崩溃率 < 0.5%

- 用户反馈响应时间 < 24 小时

## 11. 假设与约束

### 关键假设
1. **SiliconFlow API 稳定可用**：假设 SiliconFlow 服务持续稳定，API 不发生重大变更
2. **BOSS 平台可访问**：假设 BOSS 直聘 Web 端可通过技术手段自动化
3. **用户愿意提供 API Key**：假设目标用户有能力获取并愿意使用自己的 SiliconFlow API Key
4. **Windows 市场足够大**：假设 Windows 用户群覆盖了足够多的目标用户

### 技术约束
1. 仅支持 Windows 10/11
2. 需要用户自行获取 SiliconFlow API Key
3. 需要 BOSS 直聘账号（手机号注册）

### 时间约束
- V1 版本计划开发周期：4-6 周
- 核心功能优先，辅助功能迭代
